# 🔄 第9章：变换与过渡 —— 流畅的 UI 反馈

用户与界面的每一次交互，都期待即时而流畅的反馈。按钮被按下时应微微下沉，卡片悬停时可轻盈上浮，侧边栏滑出需如丝绸般顺滑——这些细微的动效，正是通过 `transform` 与 `transition` 的默契配合实现的。本章将从 `transform` 的基础函数出发，深入浏览器渲染管线的性能奥秘，剖析贝塞尔曲线的物理隐喻，并掌握 `will-change` 这一性能利器。



## 🔄 9.1 `transform` 属性全解析：平移、旋转、缩放、倾斜与矩阵

`transform` 允许你在二维或三维空间中移动、旋转、缩放和倾斜元素，而**完全不干扰文档流**——其他元素的位置不会因此重新计算。这使得它成为实现交互动画的理想选择。

### 📦 2D 变换函数速览

| 函数 | 作用 | 示例 |
|------|------|------|
| `translateX(x)` / `translateY(y)` | 水平/垂直平移 | `transform: translateX(20px);` |
| `translate(x, y)` | 二维平移 | `transform: translate(20px, -10px);` |
| `scaleX(s)` / `scaleY(s)` | 水平/垂直缩放 | `transform: scaleX(1.2);` |
| `scale(s)` / `scale(sx, sy)` | 统一/分别缩放 | `transform: scale(1.1);` |
| `rotate(deg)` | 旋转 | `transform: rotate(45deg);` |
| `skewX(deg)` / `skewY(deg)` | 倾斜 | `transform: skewX(10deg);` |
| `skew(degX, degY)` | 二维倾斜 | `transform: skew(5deg, -2deg);` |
| `matrix(a, b, c, d, e, f)` | 矩阵变换（底层统一表示） | 见下文 |

### 🧪 实战示例：悬停动画组合

```html
<button class="transform-btn">悬浮我试试</button>
```

```css
.transform-btn {
  padding: 12px 24px;
  background: oklch(0.6 0.2 260);
  color: white;
  border: none;
  border-radius: 8px;
  font-size: 16px;
  cursor: pointer;
  /* 添加过渡，让变换平滑 */
  transition: transform 0.2s ease;
}
.transform-btn:hover {
  /* 组合平移 + 缩放 */
  transform: translateY(-4px) scale(1.05);
}
.transform-btn:active {
  /* 按下时下沉 */
  transform: translateY(0) scale(0.98);
}
```

**效果**：悬停时按钮向上浮起并略微放大，点击时下沉缩小，提供清晰的按压反馈。由于只使用 `transform`，不影响周围文字布局。

### 🌀 变换的参照原点：`transform-origin`

默认情况下，旋转变换以元素中心为原点。`transform-origin` 允许你自定义参照点。

```css
.rotate-demo {
  width: 100px;
  height: 100px;
  background: #f1948a;
  transition: transform 0.3s;
}
/* 以左上角为原点旋转 */
.rotate-top-left {
  transform-origin: top left;
}
.rotate-top-left:hover {
  transform: rotate(15deg);
}
/* 以右下角为原点旋转 */
.rotate-bottom-right {
  transform-origin: bottom right;
}
.rotate-bottom-right:hover {
  transform: rotate(15deg);
}
```

`transform-origin` 也接受具体像素或百分比坐标：`transform-origin: 50px 20px;`

### 🧬 多重变换的顺序陷阱

变换函数按**书写顺序从右向左执行**。顺序不同，结果可能天差地别。

```css
/* 先平移，再旋转 */
.transform-order1 {
  transform: translateX(50px) rotate(45deg);
}
/* 先旋转，再平移 */
.transform-order2 {
  transform: rotate(45deg) translateX(50px);
}
```

**可视化差异**：
- 顺序1：元素向右移动 50px，然后绕自身新位置的中心旋转 45°。
- 顺序2：元素原地旋转 45° 后，沿着**旋转后的 X 轴**方向移动 50px（即沿 45° 对角线方向移动）。

**最佳实践**：通常将平移放在前面，缩放和旋转放在后面，以符合直觉的「先定位，后变形」。

### 🔢 矩阵变换 `matrix()`：所有 2D 变换的数学本质

`transform` 的所有 2D 函数在底层都被转换为一个 3×3 的矩阵（用 6 个参数表示）。`matrix(a, b, c, d, e, f)` 中的参数含义：

- `a`：水平缩放（scaleX）
- `b`：垂直倾斜（skewY）
- `c`：水平倾斜（skewX）
- `d`：垂直缩放（scaleY）
- `e`：水平平移（translateX）
- `f`：垂直平移（translateY）

**映射关系**：
| 函数 | 等效 matrix |
|------|-------------|
| `translate(tx, ty)` | `matrix(1, 0, 0, 1, tx, ty)` |
| `scale(sx, sy)` | `matrix(sx, 0, 0, sy, 0, 0)` |
| `rotate(θ)` | `matrix(cosθ, sinθ, -sinθ, cosθ, 0, 0)` |
| `skew(θx, θy)` | `matrix(1, tan(θy), tan(θx), 1, 0, 0)` |

```css
/* 等效写法 */
transform: translate(30px, 20px) rotate(30deg) scale(1.2);
/* 等价于一个 matrix */
```

虽然日常开发极少手写 `matrix`，但理解其存在有助于解释某些高级动画库的实现原理。

### 🚀 3D 变换速览

2D 变换是 3D 变换在 Z 轴上的特例。常用 3D 函数包括：

```css
/* 3D 平移 */
transform: translateZ(50px);
transform: translate3d(x, y, z);

/* 3D 旋转 */
transform: rotateX(45deg);
transform: rotateY(45deg);
transform: rotate3d(1, 1, 0, 45deg);

/* 3D 缩放 */
transform: scaleZ(2);
transform: scale3d(sx, sy, sz);

/* 透视（必须设置在父容器上） */
perspective: 500px;
perspective-origin: center;
```

**透视卡片翻转示例**：

```html
<div class="card-3d">
  <div class="card-inner">
    <div class="card-front">正面</div>
    <div class="card-back">背面</div>
  </div>
</div>
```

```css
.card-3d {
  width: 200px;
  height: 260px;
  perspective: 1000px;
}
.card-inner {
  position: relative;
  width: 100%;
  height: 100%;
  transition: transform 0.6s;
  transform-style: preserve-3d;
}
.card-3d:hover .card-inner {
  transform: rotateY(180deg);
}
.card-front, .card-back {
  position: absolute;
  width: 100%;
  height: 100%;
  backface-visibility: hidden;
  border-radius: 12px;
  display: flex;
  align-items: center;
  justify-content: center;
}
.card-front { background: #aed6f1; }
.card-back {
  background: #f9e79f;
  transform: rotateY(180deg);
}
```



## ⚡ 9.2 渲染性能优化：为什么只用 `transform` 和 `opacity` 做动画？

在 CSS 中实现动画，并非所有属性都「生而平等」。有些属性的变动会触发昂贵的布局重计算，导致动画卡顿掉帧；而 `transform` 和 `opacity` 则能以 60fps 丝滑运行。这背后的秘密，藏在浏览器的渲染流水线中。

### 🖥️ 浏览器渲染流水线四阶段

每次视觉变化，浏览器可能经历以下步骤：

1. **样式计算（Style）**：重新计算匹配元素的 CSS 规则。
2. **布局（Layout / Reflow）**：计算元素的几何位置和尺寸。
3. **绘制（Paint）**：将元素绘制为像素位图（填充颜色、边框、阴影等）。
4. **合成（Composite）**：将绘制好的图层合并显示到屏幕。

**关键事实**：不同 CSS 属性的变更，会触发不同阶段的重新计算，成本差异巨大。

### 📊 属性触发的渲染阶段分类

| 触发阶段 | 典型属性 | 性能代价 |
|----------|----------|----------|
| **仅合成** | `transform`、`opacity` | ⚡ 最低（跳过布局和绘制） |
| **绘制 + 合成** | `color`、`background`、`box-shadow`、`border-radius` | ⚡⚡ 中等 |
| **布局 + 绘制 + 合成** | `width`、`height`、`margin`、`padding`、`top`、`left`、`font-size` | ⚡⚡⚡ 最高 |

### 🧪 实验对比：`left` vs `translateX`

```html
<div class="move-left">使用 left</div>
<div class="move-translate">使用 translateX</div>
```

```css
.move-left {
  position: relative;
  animation: moveLeft 2s infinite;
}
@keyframes moveLeft {
  0%, 100% { left: 0; }
  50% { left: 200px; }
}

.move-translate {
  animation: moveTranslate 2s infinite;
}
@keyframes moveTranslate {
  0%, 100% { transform: translateX(0); }
  50% { transform: translateX(200px); }
}
```

**性能剖析**（使用 Chrome DevTools Performance 面板录制）：
- `left` 动画：每一帧都触发 **Layout**（绿色条）和 **Paint**（绿色条），CPU 占用高，在低端设备上容易掉帧。
- `translateX` 动画：只有 **Composite Layers**（灰色条），几乎不占用主线程 CPU，GPU 独立完成。

### 🧠 为什么 `transform` 和 `opacity` 能跳过布局和绘制？

当一个元素应用了 `transform` 或 `opacity`（且值不为 `1`），浏览器会将其提升到一个独立的**合成层（Compositor Layer）**。合成层类似于 Photoshop 中的独立图层，其内容被预先绘制为位图并上传至 GPU 显存。后续动画只需由 GPU 对图层进行平移、旋转、缩放或透明度混合，**完全无需主线程参与**。

**触发合成层的常见方式**：
- 3D 变换（`translate3d`、`rotate3d` 等）
- `will-change: transform` 或 `will-change: opacity`
- `<video>`、`<canvas>`、`<iframe>` 元素
- `backface-visibility: hidden`

### 💡 实战技巧：用 `translateZ(0)` 强制 GPU 加速

在移动端开发中，常会遇到某些低端设备动画卡顿。一个经典 HACK 是添加 `transform: translateZ(0)`，强制元素提升为合成层。

```css
.smooth-animation {
  transform: translateZ(0);  /* 或 translate3d(0,0,0) */
  will-change: transform;    /* 现代推荐做法 */
}
```

**注意**：不要滥用，每个合成层都会占用 GPU 显存。图层过多（如对整个列表项逐一添加）反而可能导致性能下降和内存溢出。

### 📈 合成层调试技巧

在 Chrome DevTools 中：
1. 打开 Rendering 标签（More tools → Rendering）。
2. 勾选 **Layer borders**，页面会显示橙色边框标记合成层。
3. 勾选 **Paint flashing**，重绘区域会闪烁绿色，帮助你识别不必要的绘制开销。



## 📈 9.3 `transition` 与贝塞尔曲线：模仿自然物理运动的缓动函数

`transition` 让属性值的变化不再是生硬的跳跃，而是平滑的插值动画。它的语法简洁，但其中的 `transition-timing-function` 却蕴含着丰富的动画哲学。

### 📝 `transition` 语法详解

```css
transition: <property> <duration> <timing-function> <delay>;
```

| 部分 | 说明 | 示例 |
|------|------|------|
| `property` | 要过渡的属性，`all` 表示所有可动画属性 | `opacity`、`transform`、`background` |
| `duration` | 过渡时长，单位 `s` 或 `ms` | `0.3s`、`200ms` |
| `timing-function` | 缓动函数，控制动画进度曲线 | `ease`、`cubic-bezier()` |
| `delay` | 延迟开始时间 | `0.1s` |

**多重过渡**：可以用逗号分隔，为不同属性设置不同参数。

```css
.card {
  transition: 
    transform 0.3s cubic-bezier(0.34, 1.56, 0.64, 1),
    opacity 0.2s ease-out,
    box-shadow 0.4s;
}
```

### 🎛️ 贝塞尔曲线（Cubic Bezier）深度解析

`cubic-bezier(x1, y1, x2, y2)` 定义了四个控制点（起始 (0,0)，终点 (1,1) 固定）形成的曲线，X 轴代表时间进度，Y 轴代表动画完成度。

**内置关键字与对应贝塞尔值**：

| 关键字 | 贝塞尔值 | 视觉特征 |
|--------|----------|----------|
| `linear` | `cubic-bezier(0, 0, 1, 1)` | 匀速，机械感强 |
| `ease` | `cubic-bezier(0.25, 0.1, 0.25, 1)` | 缓入缓出，默认值 |
| `ease-in` | `cubic-bezier(0.42, 0, 1, 1)` | 慢启动，快结束 |
| `ease-out` | `cubic-bezier(0, 0, 0.58, 1)` | 快启动，慢结束 |
| `ease-in-out` | `cubic-bezier(0.42, 0, 0.58, 1)` | 慢启动，中间快，慢结束 |

**自定义贝塞尔曲线**：通过调整两个控制点创造独特的运动节奏。

```css
/* 弹跳效果（超出再回弹） */
.bounce {
  transition: transform 0.4s cubic-bezier(0.68, -0.55, 0.27, 1.55);
}
/* 快速冲刺后急停 */
.snap {
  transition: all 0.2s cubic-bezier(0.2, 0.9, 0.1, 1.1);
}
```

**在线工具推荐**：`cubic-bezier.com` 可实时拖动控制点并生成代码，是调试缓动函数的必备神器。

### 🌟 进阶：`steps()` 阶梯函数与逐帧动画

`steps(n, <jumpterm>)` 将过渡拆分为 n 个离散步骤，适用于精灵动画或打字机效果。

```css
/* 打字机效果 */
.typewriter {
  width: 0;
  white-space: nowrap;
  overflow: hidden;
  border-right: 2px solid black;
  animation: typing 3s steps(20, end) infinite;
}
@keyframes typing {
  from { width: 0; }
  to { width: 20ch; }  /* 20 个字符宽度 */
}
```

`steps(20, end)` 表示将动画过程切割为 20 帧，每帧结束时跳变，模拟逐字出现的打字效果。

### 🌊 模仿物理运动的缓动函数设计原则

自然界中的运动极少是匀速的。优秀的 UI 动画往往借鉴物理世界的运动规律：

1. **缓出（ease-out）用于入场**：元素进入屏幕时应快启动、慢停止，模仿物体滑入并停稳。
   ```css
   .modal-enter {
     animation: slideIn 0.3s cubic-bezier(0, 0, 0.2, 1);
   }
   ```

2. **缓入（ease-in）用于退场**：元素离开时应慢启动、快消失，模仿物体被迅速移走。
   ```css
   .modal-exit {
     animation: slideOut 0.25s cubic-bezier(0.4, 0, 1, 1);
   }
   ```

3. **回弹效果（Overshoot）**：用贝塞尔曲线模拟弹性碰撞（Y 值短暂超过 1 再回落）。
   ```css
   .pop-effect {
     transition: transform 0.4s cubic-bezier(0.34, 1.56, 0.64, 1);
   }
   .pop-effect:hover {
     transform: scale(1.2);
   }
   ```



## 🧠 9.4 开发技巧：利用 `will-change` 提前告知浏览器准备合成层

`will-change` 是一个优化属性，允许开发者提前告知浏览器「某个元素的某个属性即将发生变化」，使浏览器提前进行优化准备（如创建合成层、预分配 GPU 资源）。

### 📌 基本语法

```css
.will-animate {
  will-change: transform, opacity;
}
/* 或单个属性 */
.will-animate-transform {
  will-change: transform;
}
```

### ⚠️ 使用原则：不要滥用！

`will-change` 是一把双刃剑，滥用会导致严重副作用：

1. **内存占用增加**：每个合成层都消耗 GPU 显存，过多图层可能造成移动端崩溃。
2. **浏览器优化成本**：浏览器需要预先分配资源，如果最终没有动画，反而浪费性能。
3. **作用时效有限**：优化效果通常在属性变化结束后消失，不应作为永久状态。

### ✅ 正确使用姿势

**1. 仅在动画即将发生前添加，动画结束后移除**

```javascript
const element = document.querySelector('.animate-me');
element.addEventListener('mouseenter', () => {
  element.style.willChange = 'transform';  // 提前告知
});
element.addEventListener('animationend', () => {
  element.style.willChange = 'auto';       // 及时移除
});
```

**2. 使用 CSS 类配合过渡/动画自动管理**

```css
/* 仅在 hover 或动画期间开启 will-change */
.card {
  transition: transform 0.3s;
}
.card:hover {
  will-change: transform;
  transform: scale(1.05);
}
/* 或者为持续动画的元素设置 */
.infinite-spinner {
  animation: spin 2s linear infinite;
  will-change: transform;  /* 持续动画可保留 */
}
```

**3. 仅对确实会触发布局/绘制的属性使用**

- ✅ 推荐：`transform`、`opacity`、`filter`、`backdrop-filter`
- ❌ 避免：`width`、`left`、`color` 等（无法跳过布局/绘制，优化意义不大）

### 🧪 实战对比：有 `will-change` 与无 `will-change`

```html
<div class="box no-will-change">无 will-change</div>
<div class="box with-will-change">有 will-change</div>
```

```css
.box {
  width: 100px;
  height: 100px;
  background: #aed6f1;
  transition: transform 0.2s;
}
.box:hover {
  transform: translateX(100px);
}
.with-will-change {
  will-change: transform;
}
```

**在低端 Android 设备或大量元素动画场景中**：
- 无 `will-change` 的盒子在首次 hover 时可能有轻微卡顿（浏览器临时创建合成层）。
- 有 `will-change` 的盒子提前准备好图层，动画始终丝滑。

### 🔧 `will-change` 的替代方案：`transform: translateZ(0)`

在一些老项目中，常看到 `transform: translateZ(0)` 来强制提升合成层。这是 `will-change` 普及前的 HACK，如今建议优先使用 `will-change`，因为它语义更清晰且不会引入无意义的 3D 变换。

```css
/* 旧式 HACK */
.old-hack {
  transform: translateZ(0);
}
/* 现代推荐 */
.modern {
  will-change: transform;
}
```

### 📊 DevTools 验证图层提升

1. 打开 DevTools → More tools → Rendering → **Layer borders**。
2. 有橙色边框的元素即被提升为合成层。
3. 比较有无 `will-change` 时的图层数量变化。



**本章小结**：`transform` 赋予你操纵元素空间位置的能力，`transition` 让这种变化如行云流水。理解浏览器渲染流水线是性能优化的基石——坚持用 `transform` 和 `opacity` 做动画，能让你的交互在低端设备上也保持 60fps。掌握贝塞尔曲线的设计哲学，可以让动效更贴近物理直觉；善用 `will-change` 则能未雨绸缪，为关键动画铺平道路。在下一章，我们将步入 `@keyframes` 的动画世界，探索更复杂的时间轴编排与滚动驱动动画。
