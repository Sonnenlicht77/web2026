# 🌊 第4章：文档流与格式化上下文（BFC / IFC）

在掌握了盒模型之后，下一个核心问题是：**这些盒子是如何排列的？**   

答案藏在「文档流」与「格式化上下文」中。  

文档流定义了元素默认的排列方向与规则，而 `BFC(Block Formatting Context)` 和 `IFC(Block Formatting Context)` 则是这套规则在块级和行内元素上的具体实现。理解它们，你将能从容应对浮动错乱、边距穿透、垂直对齐异常等疑难杂症。


## 📄 4.1 标准流、浮动流、定位流的视觉差异

CSS 中的「流」是指元素在页面中的排列方式。主要有三种：**标准流（Normal Flow）**、**浮动流（Float）**、**定位流（Position）**。

### 1️⃣ 标准流（Normal Flow）

标准流是页面默认的布局方式，由块级元素和行内元素共同构成。

- **块级元素**（`display: block`）：独占一行，从上到下垂直排列，宽度默认撑满父容器。
- **行内元素**（`display: inline`）：从左到右水平排列，宽度由内容决定，到达父容器边缘时自动换行。

```html
<div class="block">块级元素 1</div>
<span class="inline">行内元素 A</span>
<span class="inline">行内元素 B</span>
<div class="block">块级元素 2</div>
```

```css
.block {
  background-color: #d5e8d4;
  border: 1px solid #82b366;
  margin: 5px 0;
  padding: 8px;
}
.inline {
  background-color: #dae8fc;
  border: 1px solid #6c8ebf;
  padding: 8px;
}
```

**视觉呈现**：
- 块级元素 1 独占一行。
- 两个行内元素在同一行内从左向右排列。
- 块级元素 2 另起一行。

### 2️⃣ 浮动流（Float）

`float: left` 或 `float: right` 使元素脱离标准流，向左或向右浮动，直到碰到父容器边界或另一个浮动元素。**浮动元素不占据标准流的高度**，这会导致 **「父容器高度塌陷」**。

```html
<div class="float-container">
  <div class="float-box">浮动元素</div>
  <p class="normal-text">这是一段标准流文字。注意它会环绕浮动元素排列。</p>
</div>
```

```css
.float-container {
  border: 2px solid #b85450;
  padding: 10px;
  /* 未清除浮动，父容器高度塌陷 */
}
.float-box {
  float: left;
  width: 120px;
  height: 80px;
  background-color: #f8cecc;
  border: 1px solid #b85450;
  margin-right: 10px;
}
.normal-text {
  background-color: #f5f5f5;
}
```

**现象**：
- 浮动盒子漂浮在左侧。
- 标准流文字环绕在浮动盒子周围。
- 父容器 `.float-container` 的高度**不包含浮动元素**，边框只包裹住了文字部分（高度塌陷）。

### 3️⃣ 定位流（Position）

`position: absolute` 或 `fixed` 使元素完全脱离文档流，不再占据任何空间，后续元素会当作它不存在一样排列。

::: code-group
```html
<div class="pos-relative">
  <div class="pos-absolute">绝对定位元素</div>
  <p>标准流文字。绝对定位元素不占据空间，所以这段文字会从容器左上角开始显示。</p>
</div>
```

```css
.pos-relative {
  position: relative;
  border: 2px dashed #6c8ebf;
  height: 100px;
}
.pos-absolute {
  position: absolute;
  top: 10px;
  right: 10px;
  background-color: #e1d5e7;
  border: 1px solid #9673a6;
  padding: 5px;
}
```
:::

**现象**：
- 绝对定位元素出现在容器的右上角。
- 文字「无视」该元素的存在，直接从容器左上角开始。

### 📊 三种流的本质对比

| 特性 | 标准流 | 浮动流 | 定位流（absolute/fixed） |
|------|--------|--------|--------------------------|
| 是否占据原位置 | ✅ 占据 | ❌ 不占据（文字环绕） | ❌ 完全不占据 |
| 父容器高度是否包含 | ✅ 包含 | ❌ 不包含（需清除浮动） | ❌ 不包含 |
| 对其他元素布局影响 | 正常堆叠 | 文字环绕，块级无视 | 完全无视 |
| 典型应用场景 | 主体布局 | 图文混排、多列布局（旧） | 弹窗、悬浮按钮、下拉菜单 |



## 🧱 4.2 块级格式化上下文（BFC）的触发方法与内部特性

BFC（Block Formatting Context，块级格式化上下文）是页面上一个**独立的渲染区域**。它规定了内部块级盒子如何排列，并且**这个区域内的布局不会影响外部的元素**。

### 🔧 触发 BFC 的常用方式

| 触发方式 | 代码 | 备注 |
|---------|------|------|
| 根元素 | `<html>` | 天然 BFC |
| 浮动 | `float: left \| right` | 元素本身变成 BFC |
| 绝对/固定定位 | `position: absolute \| fixed` | |
| 行内块 | `display: inline-block` | |
| 表格单元格 | `display: table-cell` | |
| 弹性/网格容器 | `display: flex \| grid` | 直接子元素参与 flex/grid 格式化上下文，但容器本身也创建 BFC |
| `overflow` 非 `visible` | `overflow: hidden \| auto \| scroll` | 最常用的副作用触发方式 |
| **现代标准** | `display: flow-root` | **专门用于创建 BFC，无任何副作用** |

### 🧪 BFC 的三大内部特性（实战验证）

**特性一：垂直方向，块级盒子从上到下排列**

这是 BFC 最基本的规则，与标准流一致。但 BFC 保证了内部布局的独立性。

**特性二：BFC 区域不会与浮动元素重叠（自适应两栏布局）**

```html
<div style="float: left; width: 200px; height: 150px; background: #b0e0e6;">左侧浮动侧边栏</div>
<div style="overflow: hidden; height: 150px; background: #f5deb3;">右侧自适应内容（BFC）</div>
```

```css
/* 右侧通过 overflow: hidden 触发 BFC */
```

**效果**：右侧内容**不会**跑到左侧浮动元素的下方，而是自动形成两栏布局，右侧宽度自适应。

**原理**：BFC 的区域不会与浮动盒子重叠。利用这个特性，可以轻松实现**左侧固定宽度、右侧自适应**的经典布局，无需设置 `margin-left`。

**特性三：BFC 可以包含浮动元素（清除浮动）**

还记得 4.1 中父容器高度塌陷的问题吗？将父容器触发为 BFC，它会自动包裹内部的浮动元素。

```html
<div class="bfc-clearfix" style="overflow: hidden; border: 2px solid #82b366;">
  <div style="float: left; width: 100px; height: 80px; background: #d5e8d4;">浮动元素</div>
  <div style="float: left; width: 100px; height: 80px; background: #d5e8d4;">浮动元素</div>
</div>
<!-- 不需要额外的 clearfix 元素！ -->
```

**原理**：计算 BFC 高度时，内部的浮动元素也会被考虑在内。这是比 `clear: both` 更优雅的清除浮动方案。

**特性四：BFC 可以阻止外边距塌陷（隔离边距）**

3.4 节提到的父子边距穿透问题，可以通过将父元素触发为 BFC 来解决。

```html
<div style="display: flow-root; background: #e8daef;">
  <div style="margin-top: 40px; background: #d7bde2;">子元素 margin-top 被隔离在 BFC 内部</div>
</div>
```

**对比**：若父元素未触发 BFC，子元素的 `margin-top` 会穿透，与父元素的外部边距合并。


## 📝 4.3 行内格式化上下文（IFC）与 `vertical-align` 的那些坑

IFC（Inline Formatting Context，行内格式化上下文）是行内元素、行内块元素、文本节点的排列规则。它由**行盒（Line Box）** 概念主导。

### 🔤 IFC 的基本规则

- 行内元素在一行内水平排列，直到填满行宽后换行。
- 一行中的所有内容被包裹在一个虚拟的「行盒」中，行盒的高度由内部元素的 `line-height`、`vertical-align` 等属性共同决定。
- 每一行的行盒垂直堆叠，构成整个段落的高度。

### 🕳️ `vertical-align` 的经典巨坑

`vertical-align` 用于控制**行内元素或表格单元格**在垂直方向上的对齐方式。**但它只作用于行内元素和行内块元素，对块级元素无效。**

**坑点一：行内块元素底部莫名多出几像素间距**

```html
<div class="image-container">
  <img src="https://via.placeholder.com/100" alt="placeholder">
</div>
```

```css
.image-container {
  border: 2px solid #e74c3c;
  /* 注意：父容器高度可能比图片多出 3-5px */
}
img {
  width: 100px;
  height: 100px;
  /* 未设置 vertical-align */
}
```

**现象**：图片下方与容器底部之间存在几像素的空白间隙。

**原因**：`<img>` 默认是行内元素（`inline`），它坐在文本的**基线（baseline）** 上。基线下方为字母下部（如 `g`、`p` 的尾巴）预留了空间，这就是那个多余间隙的来源。

**解决方案（三选一）**：

```css
/* 方案一：将图片改为块级 */
img {
  display: block;
}

/* 方案二：修改 vertical-align 为 bottom / middle / top */
img {
  vertical-align: bottom;
}

/* 方案三：设置父容器 font-size: 0（同时影响内部文字，慎用） */
.image-container {
  font-size: 0;
}
```

**坑点二：同一行内，不同字号的文字垂直对齐错乱**

```html
<p>
  <span class="large">大号文字</span>
  <span class="small">小号文字</span>
</p>
```

```css
.large { font-size: 40px; }
.small { font-size: 16px; }
```

**默认效果**：两个 `<span>` 的**基线对齐**，导致大号文字下方露出大量空白，小号文字仿佛悬空。

**修复**：

```css
.large, .small {
  vertical-align: middle;  /* 中线对齐，视觉更和谐 */
}
```

### 📐 `line-height` 与内容区、行内盒的关系

- **内容区（Content Area）**：由字体字号和字体度量决定，`background` 默认填充此区域。
- **行内盒（Inline Box）**：由 `line-height` 决定其高度。当 `line-height > 字号` 时，多出的部分作为「行间距」（半间距）均分在文字上下。
- **行盒（Line Box）**：包裹一行中所有行内盒的最小矩形，其高度由最高的行内盒决定。

理解这三层结构，是彻底驾驭垂直对齐问题的关键。

## ☯️ 4.4 一种哲学：文档流是一切布局错乱的最终解药

许多开发者在遇到布局错乱时，第一反应是「加 `position: absolute`」「上 `float`」「硬设宽高」。这些手段短期有效，长期却让 CSS 变成一锅无法维护的粥。

**核心心法**：**先理解标准流的意图，再用 BFC / IFC 规则修复偏差，最后才考虑脱离文档流。**

#### 🧭 调试布局的思维框架

当你面对一个错乱的布局时，按以下顺序自问：

1. **它是否还在标准流中？**
   - 若被 `float`、`absolute` 拉出流外，它的位置由 `top/left` 决定，后续元素会无视它。
   - 这是导致「元素重叠」「父容器高度为零」的根源。

2. **外边距是否发生了塌陷？**
   - 检查父子边距是否合并，兄弟边距是否取最大值。
   - 触发 BFC 可一劳永逸。

3. **行内元素的垂直对齐是否符合预期？**
   - 检查 `vertical-align` 是否为默认的 `baseline`。
   - 调整 `vertical-align` 或改变 `display`。

4. **是否无意中创建了新的 BFC 或层叠上下文？**
   - 例如 `overflow: hidden` 在裁剪内容的同时也创建了 BFC，可能改变布局行为。

### 🌰 一个贯穿性的修复案例

**需求**：实现一个商品卡片，左侧是图片，右侧是标题和描述。卡片底部有操作按钮。

**错误实现**（大量 `absolute` 和固定高度）：

```css
.card {
  position: relative;
  height: 120px; /* 硬编码高度，内容多了就溢出 */
}
.card img {
  position: absolute;
  left: 10px;
  top: 10px;
}
.card .content {
  margin-left: 120px;
}
.card .button {
  position: absolute;
  bottom: 10px;
  right: 10px;
}
```

**问题**：内容长度不可控，硬编码高度导致文字溢出或大片空白。

**文档流思维的重构**：

```html
<div class="card">
  <img src="thumb.jpg" alt="商品图">
  <div class="info">
    <h3>商品名称</h3>
    <p>描述文字，可能很长也可能很短。</p>
    <button>购买</button>
  </div>
</div>
```

```css
.card {
  display: flex;          /* 启用 Flex 流，子元素回归标准流规则 */
  gap: 15px;
  padding: 15px;
  border: 1px solid #ddd;
}
.card img {
  width: 100px;
  height: 100px;
  object-fit: cover;
  flex-shrink: 0;         /* 防止图片被压缩 */
}
.info {
  flex: 1;                /* 占据剩余空间 */
  display: flex;
  flex-direction: column;
  justify-content: space-between; /* 按钮自然沉底 */
}
button {
  align-self: flex-end;   /* 按钮靠右 */
}
```

**优势**：
- **无固定高度**：卡片高度由内容撑开，内容再多也能自适应。
- **无绝对定位**：所有元素都在文档流（Flex 流）中，位置由弹性规则决定，相互呼应。
- **易于维护**：修改图片大小、增加描述行数，布局自动调整，无需重算偏移量。

### 🧠 结语

文档流不是束缚，而是秩序。当你敬畏它、理解它，CSS 布局便从「猜谜游戏」变为「逻辑推演」。BFC 是块级世界的结界，IFC 是行内世界的法则，而 Flex 和 Grid 则是现代布局对文档流的优雅扩展。在下一章，我们将正式踏入 Flexbox 的世界，探索一维弹性布局的全部秘密。
